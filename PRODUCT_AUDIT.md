## 2. Пользовательские сценарии

### Сценарий A: «Впервые пришёл — понял задачу — получил результат» (основной)

```
Точка входа (SEO/_direct)
    → Главная (лендинг утилит)
    → Выбирает категорию или ищет
    → Переходит на страницу инструмента
    → Вводит данные
    → Получает результат
    → Копирует / скачивает / переходит к связанному инструменту
```

**Примеры**: VAT Calculator → копирует результат; QR Code Generator → скачивает PNG; Number to Words → вставляет в документ.

### Сценарий B: «Пришёл из статьи / по ссылке»

```
Внешняя ссылка / статья / Telegram
    → Страница статьи
    → Кликает на связанный инструмент (CTA в article.toolSlug)
    → Использует инструмент
```

**Пример**: Статья о НДС → ссылка на VAT Calculator → пользователь считает НДС.

### Сценарий C: «Ищу нужный инструмент»

```
Главная → Поиск (header)
    → Вводит запрос
    → Видит результаты поиска
    → Переходит на инструмент
```

Альтернатива: категориальная фильтрация через `?category=calculators` на главной.

### Сценарий D: «Повторное использование инструмента»

```
Возвращается на инструмент (bookmark / память)
    → Данные восстановлены из localStorage
    → Меняет параметры
    → Получает новый результат
```

**Пример**: Compound Interest — вернулся, изменил ставку, увидел новый график.

### Сценарий E: «Аудит чужого сайта»

```
SEO Audit Pro
    → Вводит URL
    → Видит score, приоритеты, breakdowns
    → Может открыть raw data
    → Может скопировать данные
```

### Сценарий F: «Читает статью»

```
Articles Index
    → Выбирает статью
    → Читает контент
    → Переходит по CTA к инструменту
    → Видит переключалку RU/EN
```

### Сценарий G: «Обратная связь»

```
Footer → "Написать нам"
    → Заполняет форму (имя, email, сообщение)
    → Отправляет через Feedback Worker
    → Видит подтверждение или ошибку
```

## 4. Потенциальные слабые места UX и логики переходов

### 4.1 Навигация и маршрутизация

**A. Нет breadcrumbs на статьях**
Статья имеет путь `/ru/articles/slug`, но breadcrumb показывает только "Статьи", без текущей статьи. Пользователь не видит глубокую навигацию.

**B. Language switcher на статьях может сломать контекст**
Переключение языка на статье `/ru/articles/kak-sdelat-qr...` → `/en/articles/` — но EN-версия может называться иначе. Текущая реализация (`translationKey` → slug lookup) может не найти перевод, если slug отличается. Не проверено, что переключение языка работает корректно для всех статей.

**C. Фильтр категории на главной (`?category=`) — не отображается в UI**
URL меняется, но пользователь не видит активный фильтр на странице. Нет визуального индикатора «Показаны только калькуляторы».

**D. Feedback — изолированная страница без связи с инструментами**
Форма обратной связи доступна только из Footer. Нет фидбэк-линка с конкретного инструмента. Если у пользователя проблема с калькулятором НДС, он не может сообщить об этом с контекстом.

### 4.2 Состояния и edge-cases

**E. Пустое состояние на ArticlesIndex**
Если Worker API не вернул статей, показывается пустой экран с текстом "Статьи появятся позже". Нет иллюстрации, нет CTA, нет кнопки «Попробовать ещё раз».

**F. SEO Audit Pro — timeout без retry**
Если Worker не ответил за 12 секунд, пользователь видит ошибку. Нет кнопки «Попробовать снова» рядом с ошибкой (только общая форма).

**G. QR Code Generator — silent fail при невалидном URL**
Если пользователь ввёл невалидный URL для QR-кода типа "data URL" или слишком длинный контент — генерация может тихо упасть без сообщения пользователю. Есть `generationError` state, но не проверено, что он отображается в UI.

**H. NumberToWords — очень длинные числа**
При вводе числа > 10¹⁵ результат может быть нечитаемым или обрезаться. Нет валидации на верхнюю границу.

**I. URL Shortener — URL добавляет https:// автоматически**
Если пользователь ввёл `http://example.com`, он получит `https://https://example.com`. Это сломает ссылку. Нужно проверять: если уже есть протокол — не добавлять `https://`.

**J. Password Generator — strength label может не перевестись**
Если `strengthResult.label` не совпадает с ключом из `strengthLabels`, пользователь увидит английский label вместо переведённого. `|| strengthResult.label` fallback не гарантирует локализацию.

### 4.3 Логика переходов

**K. CompoundInterest использует LineChart (canvas) без chart.js**
Уже хорошо — это свой компонент. Но GraphPanel (Chart.js) грузится через lazy даже когда не используется (Calculator по умолчанию в режиме `calculator`, не `graph`). Это правильно, но GraphPanel может долго загружаться при первом переключении на режим графика.

**L. "Скопировано" — нет визуального фидбека после копирования**
CopyButton показывает "Скопировано" на 2 секунды, но после этого не возвращает состояние автоматически в некоторых реализациях. Не проверено, что feedback устойчив.

**M. MetaTagsGenerator — автосохранение в localStorage каждые 1 сек**
При быстром наборе текста это создаёт множество write-operations. Не критично, но можно debounce до 2 сек или использовать useDeferredValue.

**N. Calculator — history Panel не очищается при переключении языка**
Если пользователь набрал историю в RU, переключил на EN — история остаётся на русском языке (выражения на русском). Это может быть ожидаемым поведением, но неочевидно.

### 4.4 Визуальные и интерактивные проблемы

**O. QRCodeGenerator — слишком много опций на экране**
7 типов QR (text, URL, email, phone, SMS, WiFi, vCard) + десятки настроек внешнего вида + логотип + прозрачность + формат экспорта. Пользователь может быть перегружен. Рекомендуется wizard flow или progressive disclosure.

**P. SEOAuditPro — raw data technical snapshot**
Опция «Raw audit data» раскрывает внутренний JSON от Worker. Интерфейс непонятен нетехническому пользователю. Нужно либо скрыть, либо добавить explainer.

**Q. DateDifferenceCalculator — сложный интерфейс**
4 режима (Date, Time, Countdown, Age), 10+ полей. Можно упростить через tabbed interface, а не list всех режимов.

**R. Password strength indicator — линейная шкала**
Бар силы пароля визуально простой. Нет объяснения, почему пароль «слабый» — нет breakdown факторов.

### 4.5 Проблемы с внешними зависимостями

**S. URL Shortener зависит от is.gd**
Если is.gd упадёт или изменит API, пользователь не получит результата. Нет fallback на другой сервис и нет информативного сообщения при ошибке is.gd.

**T. Feedback Worker может быть недоступен**
Если `apifeedback.qten.workers.dev` недоступен — timeout 12 сек, ambiguous delay 3.5 сек. После этого пользователь видит «pending» состояние, которое непонятно. Нет чёткого «Попробуйте позже или напишите на email».

**U. SEO Audit Pro — CORS error в браузере**
Если API-запрос блокируется CORS, пользователь видит общее сообщение об ошибке. Не объясняется, что это политика браузера и как обойти.

## 6. Пользовательские потоки

### Поток 1: «Бесплатный инструмент → результат»

```
[Search / SEO] → Главная → Выбрал инструмент
    → Ввёл данные → Получил результат
    → Скопировал / Скачал → Задача выполнена
    → Может: перейти к связанному → Цикл продолжается
```

**Застревания**: главная страница не даёт понять ценность конкретного инструмента без пробы.

### Поток 2: «Читатель → пользователь инструмента»

```
[Внешняя ссылка / Статья в Telegram] → Статья
    → Видит CTA "Попробовать [инструмент]"
    → Переходит на страницу инструмента
    → Использует → Результат
    → Читает related articles
```

**Застревания**: CTA в статье ведёт на `/ru/tool-slug`, но если статья EN, а инструмент не переведён — возникает языковый дисбаланс.

### Поток 3: «Повторный визит»

```
[Bookmark / Запомнил URL] → Инструмент
    → Данные восстановлены из localStorage
    → Изменил параметры → Новый результат
    → Скопировал / Завершил
```

**Застревания**: localStorage может содержать устаревшие данные, если структура инструмента изменилась.

### Поток 4: «Проблема → обратная связь»

```
[Использует инструмент] → Что-то пошло не так
    → Ищет "написать нам" в footer
    → Заполняет форму (имя, email, сообщение)
    → Отправил → Не понимает, ушло ли
```

**Застревания**: нет подтверждения с ID тикета, нет email-уведомления, нет статуса.
